1. Apertura sugerida

"Nuestro sistema se llama SIGOMEI y resuelve la gestion de mantenimiento industrial. Lo implementamos como una aplicacion distribuida: el cliente JavaFX no accede directamente a la logica de negocio, sino que invoca un servicio remoto publicado por el servidor mediante Java RMI. La capa de servicio concentra validaciones, reglas de negocio y operaciones CRUD sobre equipos, tecnicos y ordenes."

2. Arquitectura distribuida implementada

Cliente

Aplicacion JavaFX: SigomeiFxApp. Presenta pantallas, captura datos y llama metodos remotos del contrato SigomeiRemote.

No implementa reglas centrales: si el usuario deja campos vacios, captura el error y muestra un dialogo; si la regla depende del negocio, espera respuesta del servidor.

Servidor

SigomeiServer crea el registry RMI en el puerto configurado, instancia SigomeiRemoteImpl y publica el servicio con el nombre SIGOMEI.

SigomeiRemoteImpl extiende UnicastRemoteObject y delega en UsuarioService, EquipoService, TecnicoService y OrdenService.

3. Justificacion tecnica de RMI

  • Encaja con Java a Java: cliente, contrato, DTOs y servidor estan hechos en Java; RMI permite invocar metodos remotos casi como metodos locales.
  • Contrato tipado: SigomeiRemote obliga a usar tipos concretos como EquipoDTO, TecnicoDTO, OrdenDTO y excepciones de dominio.
  • Menos protocolo manual: con sockets habria que disenar mensajes, serializacion, codigos de operacion, parseo y manejo de errores. RMI ya resuelve transporte, stubs y serializacion de objetos.
  • Separacion clara: el cliente no conoce repositorios ni reglas internas; solo conoce la interfaz remota.
  • Trade-off: RMI acopla mas a Java que sockets o HTTP, pero para este proyecto academico con ambos extremos en Java reduce complejidad y hace mas clara la demostracion distribuida.

4. Contrato de comunicacion cliente-servidor

El contrato esta en SigomeiRemote, que extiende Remote. Cada operacion declara RemoteException y, cuando aplica, excepciones de dominio: ValidacionException, ReglaNegocioException, RegistroNoEncontradoException y AutenticacionException.

8. Demostracion en vivo recomendada

A. CRUD remoto de 3 entidades

  1. Levantar servidor: mvn exec:java si tienen plugin, o ejecutar SigomeiServer desde el IDE.
  2. Abrir cliente JavaFX SigomeiFxApp y entrar con admin / admin123.
  3. Equipo: registrar un equipo nuevo, filtrarlo, editar ubicacion o estado.
  4. Tecnico: registrar tecnico nuevo, editar telefono, eliminarlo si no tiene ordenes.
  5. Orden: registrar orden con equipo y tecnico compatibles, actualizar descripcion, cambiar de programada a en ejecucion.

9. Analisis critico, defectos y recomendaciones

Defectos/riesgos detectados

  • El caso CP-35 queda manual porque depende de apagar el servidor durante una operacion.
  • La persistencia de prueba usa memoria; para produccion conviene integrar mas pruebas con MySQL.
  • La GUI atrapa excepciones generales; podria diferenciar mejor validacion, regla de negocio y conexion.

Valor de TDD

  • Obligo a documentar reglas como comportamiento verificable.
  • Evito regresiones al modificar servicios.
  • Hizo clara la matriz RF/RN - caso - estado.
Recomendaciones: automatizar el caso de desconexion con pruebas de integracion controladas, agregar validaciones visuales previas en JavaFX, separar repositorios reales de repositorios en memoria mediante interfaces, y mantener JaCoCo como evidencia en cada entrega.

10. Preguntas tecnicas probables y respuestas

PreguntaRespuesta sugerida
Por que eligieron RMI y no sockets?Porque ambos extremos estan en Java. RMI nos dio invocacion remota tipada, serializacion de DTOs y manejo de excepciones sin definir un protocolo manual. Sockets darian mas control, pero tambien mas codigo de bajo nivel.
Cual es el contrato de comunicacion?La interfaz SigomeiRemote. Define las operaciones que el cliente puede invocar, los DTOs que viajan y las excepciones que puede recibir.
Que viaja por la red?DTOs y parametros simples: IDs, fechas, catalogos, textos y costos. No viajan objetos de servicio ni repositorios.
Donde estan las reglas de negocio?En la capa service, especialmente OrdenService, EquipoService y TecnicoService. El cliente solo solicita operaciones y muestra resultados o errores.
Como aplicaron TDD?Primero se definieron pruebas positivas y negativas para reglas y validaciones. Luego se implemento lo minimo en servicios para pasar rojo a verde, y despues se ejecuto la suite completa con JaCoCo.
Que significa ROJO y VERDE?ROJO: la prueba falla porque la funcion aun no cumple la regla. VERDE: se implementa la logica y la prueba pasa.
Como saben que el CRUD remoto funciona?El contrato remoto expone CRUD para Equipo, Tecnico y Orden; CrudServiceTest verifica esas operaciones en servicios y la demo JavaFX las invoca desde cliente contra servidor RMI.
Que pasa si el servidor no esta disponible?La llamada RMI falla con una excepcion de conexion/remota. Existe CP32 para servidor no disponible y CP35 como prueba manual para desconexion durante operacion.
Por que usar DTOs?Porque separan datos de transporte de la logica interna. El cliente recibe estructuras simples y no conoce implementaciones del servidor.
Que regla consideran mas importante?RN-02 y RN-06: evitan duplicidad de ordenes activas y obligan a cerrar una orden con datos completos, lo que protege consistencia operativa.
La base de datos en memoria reemplaza MySQL?No. Sirve para pruebas unitarias rapidas y repetibles. MySQL es la dependencia real o futura para persistencia; en TDD conviene aislar reglas primero.
Que cubre JaCoCo?Esta configurado para cubrir com/sigomei/servidor/service/*, es decir, la capa donde estan reglas y validaciones principales.
Que limitacion tiene RMI?Acopla el sistema a Java y no es tan interoperable como HTTP/REST. En este proyecto conviene porque simplifica la arquitectura distribuida Java-Java.
Como se reparte la participacion del equipo?Una persona explica arquitectura, otra contrato RMI, otra TDD/cobertura, otra demo CRUD, otra reglas de negocio y todos contestan al menos una pregunta tecnica.
